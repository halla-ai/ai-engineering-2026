## What is Ralphthon?

**Ralphthon** is the capstone project for this course. Working **individually**, you will design and implement an **autonomous agentic system that solves a real software problem**, based on the Ralph Loop methodology.

The name is borrowed from the industry term "Hackathon." As the Ralph Loop methodology spread in early 2026, **Ralphthon** culture emerged — and this is your chance to experience it firsthand.

<Aside type="note" title="Week 8 proposal → Week 13 design document">
The big picture of your topic and architecture was already locked in during the [Week 8 project proposal](/en/weeks/week-08). Week 13 is about expanding that into an **implementation-ready detailed design** — per-agent interfaces, state file schemas, MCP tool lists, test plans.
</Aside>

## Recommended Project Ideas

1. **Autonomous Code Review System** — Automatically reviews GitHub PRs and suggests improvements
2. **Automatic Test Generator** — Generates pytest tests automatically from an existing codebase
3. **Documentation Automation** — Automatically updates README and API docs when code changes
4. **Automatic Bug Fixer** — Reads GitHub Issues and automatically finds and fixes bugs
5. **Performance Optimization Agent** — Analyzes profiling results and automatically fixes performance bottlenecks

## Architecture Design Checklist

<Steps>
1. **Problem Definition** — Clearly define the real problem to solve and the criteria for success

2. **Agent Role Design** — How many agents? What are each agent's roles and responsibilities?

3. **Data Flow Design** — Define the artifact formats passed between agents

4. **Harness Design** — Backpressure mechanisms, garbage collection policies

5. **Infrastructure Design** — Which MIG slices? Which models?

6. **Evaluation Method** — How will the system's performance be measured?
</Steps>

<Aside type="note" title="Design Document Requirements">
The architecture design document submitted as the Week 13 assignment must include an agent interaction diagram, a data flow diagram, and a rationale for technology stack choices.
</Aside>
